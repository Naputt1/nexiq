# Plan: Token Reduction Benchmarking

This plan outlines a strategy to quantify the token efficiency and performance gains of the MCP-centric architecture compared to traditional file-based exploration.

## 1. Objectives
- **Quantify Token Savings**: Compare total input/output tokens used to solve specific code-discovery tasks.
- **Measure Latency**: Compare time-to-success for complex queries.
- **Cache Impact**: Compare "Cold Start" (first analysis) vs "Warm Start" (cached graph) performance.
- **Model Variability**: Measure performance across different state-of-the-art LLMs (GPT-4o, Claude 3.5 Sonnet, Gemini 1.5 Pro).
- **Baseline Comparison**: Use standard filesystem MCP tools (read_file, grep, list_dir) as the control group.

## 2. Benchmark Scenarios & Project Sizes

Tasks will be performed across three project tiers to measure how token savings scale with complexity:

### Project Tiers
| Tier | Size (LOC) | Characteristics |
| :--- | :--- | :--- |
| **Small** | < 5,000 | Simple component structure, few props, no external state management. |
| **Mid** | 20,000 - 30,000 | Includes React Router, Zustand/Redux, and modular components with 2-3 levels of nesting. |
| **Large** | 50,000 - 200,000+ | Monorepo/Multi-module, complex routing, heavy state management, optimization libs (TanStack Query), and large UI libraries. |

### Target Models
The benchmark will be executed using the latest versions of:
- **OpenAI**: GPT-5.2 Codex
- **Anthropic**: Claude 4.6 Sonnet / 4.5 Haiku
- **Google**: Gemini 3 Flash Preview
- **DeepSeek**: DeepSeek v3.2
- **Meta (Optional)**: Llama 3.1 405B (via Groq/OpenRouter)

### Scenarios
Scenarios will be structurally identical across project tiers but adapted to use actual components/hooks found in each project:

1.  **Scenario 1: Global Symbol Discovery (Breadth)**
    - **Prompt**: "Find the definition of `[PrimaryComponent]` and list every file where it is rendered."
    - **Goal**: Measure tokens used to map a component's presence across the entire codebase.

2.  **Scenario 2: Prop Analysis & Data Trace (Depth)**
    - **Prompt**: "Identify all props accepted by `[SharedComponent]` and find exactly where the data for the `[DataProp]` prop originates in `[ParentComponent]`."
    - **Goal**: Measure tokens used to traverse parent-child relationships and analyze interface contracts.

3.  **Scenario 3: Dependency Impact Analysis (Complexity)**
    - **Prompt**: "Find all usages of the `[CoreHook]` hook and list every component that depends on the `[NestedField]` field of the returned object."
    - **Goal**: Measure tokens used to track state/logic propagation through the component tree.

## 3. Test Methodology

### Fail-Safe Mechanism
- **Tool Call Cap**: Each scenario will have a hard limit of **10 tool calls**. If reached without a definitive answer, the test is marked as `FAILED (LOOP DETECTED)`.
- **Confidence Threshold**: The LLM must explicitly state "I have found the answer" to stop the clock.
- **Fallback Rule**: If `nexiq` returns an empty or suspicious result (e.g., missing essential file paths), the LLM must attempt one alternative query before terminating the test.

### Approach A: Baseline (Standard MCP)
- Use only `read_file`, `grep_search`, and `list_directory`.
- Record every tool call and the size of the returned text content.

### Approach B: React Map (No Cache / Cold)
- Delete `.nexiq/cache`.
- Call `open_project` followed by `get_symbol_info` or `list_files`.
- Measure analysis time + tool output tokens.

### Approach C: React Map (Cached / Warm)
- Run tasks on a pre-analyzed project.
- Measure tool output tokens and latency.

## 4. Metrics to Collect
- **Total Request Tokens**: Prompt tokens sent to the LLM.
- **Total Response Tokens**: Tokens generated by the LLM.
- **Tool Payload Tokens**: Total tokens returned by all tool calls combined.
- **Total Tool Calls**: Number of round-trips required to reach the answer.
- **Execution Time**: Total wall-clock time from first prompt to final answer.

## 5. Implementation Phases

### Phase 1: Benchmark Utility
- Create a `scripts/benchmark.ts` that:
    - Clears/restores cache.
    - Intercepts and logs tool call sizes.
    - Uses a tokenizer (e.g., `tiktoken`) to provide exact counts.

### Phase 2: Scenario Execution
- Run each scenario 3 times per approach to calculate averages.
- Log results to `benchmark-results.json`.

### Phase 3: Reporting
- Generate a Markdown report comparing:
    - % Token reduction (Baseline vs Warm) per model.
    - Model-specific "Reasoning Efficiency" (Tool calls vs Success).
    - Time saved per model and project size.
    - "Token Efficiency Score" (Success / Total Tokens).
